DigitalBonesBox: Complete Project Guide

Everything you need to know about the project structure and how each file works

What You'll Learn

This guide breaks down every important file in the DigitalBonesBox project and explains what it does in simple terms. By the end, you'll understand how a real web application is built and organized.

1. Project Overview: What is DigitalBonesBox?

DigitalBonesBox is an interactive web application that lets users learn about human anatomy - specifically the bony pelvis (hip bones). Think of it like an interactive textbook where you can:

  • Search for specific bones by typing their name
  • Click on bones to see detailed images
  • Read descriptions about bone functions and landmarks
  • Filter bones by category or body part
  • Learn from interactive tutorials and studies
Key Concept: The application is split into two parts:
  • Frontend: What you see in your browser (HTML, CSS, JavaScript)
  • Backend: The server that stores and sends data (Node.js)

2. Frontend Files (What You See)

The frontend is everything the user interacts with in their browser. These files control the look, feel, and behavior of the application.

Main HTML File

boneset.html HTML

What it does: This is the main HTML file - the foundation of the entire application. It's like the blueprint of a building.

Contains:

  • Header/Navigation: The top bar with "Home", "Tutor", "Study", "Help" buttons
  • Search Bar: Where users type bone names to search
  • Sidebar: Navigation menu that appears on the left side
  • Main Content Area: Where bone images and descriptions appear
  • Login/Sign Up: User authentication section

Why it matters:

Without HTML, there's nothing to display. This file defines the structure of every element the user sees.

Styling File

style.css CSS

What it does: Controls the appearance and layout of the application. It makes the HTML look professional.

Controls:

  • Colors: Blue headings, gold accents, white backgrounds
  • Fonts: Font sizes, styles, and weights
  • Spacing: Margins, padding, and alignment
  • Layout: How elements are positioned on the page (flexbox, grid)
  • Responsive Design: How the app looks on phones vs. computers
  • Animations: Smooth transitions and hover effects

Why it matters:

CSS turns boring text and buttons into a beautiful, professional-looking application. The user experience depends on good CSS.

JavaScript Files (The Brain)

These files contain the logic that makes the app interactive. They handle clicking, typing, searching, and updating the display.

main.js JavaScript

What it does: The central control center. This is where the core application logic happens.

Handles:

  • Initializing the application when the page loads
  • Loading bone data from the server
  • Updating the display when users interact with the app
  • Managing the app state (what's currently selected, filtered, etc.)
  • Coordinating between other JavaScript files

Example workflow:

  1. User opens the app → main.js loads
  2. main.js asks the server for bone data
  3. Server responds with bone information
  4. main.js displays the bones in the sidebar
  5. User clicks a bone → main.js updates the image and description
api.js JavaScript

What it does: Handles communication between the frontend and backend server. It's like a messenger.

Key functions:

  • fetch(): Requests data from the server (e.g., "Give me all bones")
  • Error handling: Deals with problems if the server doesn't respond
  • Data formatting: Converts server responses into usable JavaScript objects

Example request:

// Ask the server for all bones
await api.getBones();

// Ask the server for a specific bone
await api.getBoneById('ischial_tuberosity');
search.js JavaScript

What it does: Powers the search functionality. When users type in the search bar, this file filters the bones and shows matching results.

Features:

  • Real-time filtering as users type
  • Highlighting matching text
  • Handling special characters and typos
  • Performance optimization for large datasets
dropdowns.js JavaScript

What it does: Manages dropdown menus and filters. Users can filter bones by category (e.g., "Anterior vs. Posterior").

Manages:

  • Dropdown menu opening/closing
  • Filtering bones based on selected category
  • Sorting options
  • Keyboard accessibility (arrow keys, Enter)
viewer.js JavaScript

What it does: Controls how bone images are displayed. When a user clicks on a bone, this file handles showing the image and related information.

Handles:

  • Loading and displaying bone images
  • Image zoom and pan functionality
  • Displaying bone descriptions and landmarks
  • Switching between different views (anterior, posterior, etc.)

3. Backend Files (Where Data Lives)

The backend is the server that stores all the bone data and responds to requests from the frontend. It's where the "brain" of the application runs.

server.js Node.js / Backend

What it does: This is the server - a program that runs 24/7 and responds to requests from the frontend.

Key responsibilities:

  • Receives requests: "Give me all bones" or "Give me ischial tuberosity"
  • Fetches data: Gets bone information from GitHub or a database
  • Security: Prevents malicious requests and validates input
  • Performance: Caches data to respond faster
  • Sends responses: Returns bone data in JSON format
  • Error handling: Responds with error messages if something goes wrong

Example flow:

Frontend asks: "Get all bones"
Server checks cache (fast memory storage)
If not cached, fetch from GitHub
Validate and format the data
Send JSON response back to frontend
Frontend displays the bones

Key Server Endpoints (API)

An "endpoint" is a specific request the frontend can make. Here are the main ones:

Endpoint What It Does Example Response
/api/bones Returns all bones in the system Array of all bone objects
/api/bones/:id Returns a specific bone by ID Single bone object with details
/api/bones/:id/image Returns the image for a specific bone Image file or URL
/api/search?q=query Searches for bones by name Filtered array of matching bones
/api/categories Returns all bone categories/filters List of category names

4. How It All Works Together

User opens the app:
boneset.html loads in the browser
style.css applies styling (blue colors, layout, fonts)
main.js initializes and starts running
api.js sends a request to the server
server.js receives the request and sends back bone data
main.js receives the data and calls sidebar functions
App displays list of bones - user can now interact
User searches for "ischial":
User types in search bar
search.js detects the input event
search.js filters bones matching "ischial"
style.css highlights matching results
User sees filtered list with 3 matches
User clicks on "Ischial Tuberosity":
User clicks on bone name
main.js detects the click
api.js requests detailed information from server
server.js sends bone data and image URL
viewer.js displays the image
User sees bone image, description, and anatomical landmarks

5. Development Setup: How to Run It

Requirements:

  • Node.js installed (download from nodejs.org)
  • npm (comes with Node.js)
  • A code editor (VS Code recommended)
  • Git (for version control)
  • Live Server extension for VS Code

Steps to Run:

Step 1: Install Live Server Extension in VS Code

Search for "Live Server" by Ritwick Dey in the VS Code Extensions marketplace and install it

Step 2: Navigate to the backend folder
cd DigitalBonesBox/boneset-api
Step 3: Install dependencies
npm i
Step 4: Start the backend server
npm start

This starts the Node.js server. Keep this terminal running!

Step 5: Open boneset.html with Live Server

In VS Code, navigate to DigitalBonesBox/templates/boneset.html, right-click on the file, and select "Open with Live Server"

Step 6: Open in browser

Live Server will automatically open your browser at http://localhost:5500 (or the next available port)

The frontend will connect to the backend server running on http://localhost:8000

7. Important Concepts to Understand

API (Application Programming Interface)

An API is a set of rules that allows two programs to communicate. In this app:

  • Frontend uses the API to ask the backend for data
  • Backend defines the API with specific endpoints
  • Communication uses HTTP (the internet language)
  • Data is formatted as JSON (easy to read for computers)

JSON (JavaScript Object Notation)

JSON is a simple format for storing and sending data. Example:

{
  "id": "ischial_tuberosity",
  "name": "Ischial Tuberosity",
  "region": "posterior",
  "description": "The pointed projection of the ischium where..."
}

Caching

Storing data in fast memory so it doesn't need to be fetched repeatedly. In server.js, bone data is cached so the second request is instant instead of waiting for GitHub.

Responsive Design

Making the app look good on phones, tablets, AND computers. The CSS uses media queries to adjust layouts based on screen size.

Event Listeners

JavaScript code that "listens" for user actions (clicking, typing, scrolling) and responds. Example: "When user types in search box, run the search function."
