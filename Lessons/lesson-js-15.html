JavaScript Best Practices

Lesson 15 of 15

What You'll Learn

In this final JavaScript lesson, you will learn professional best practices that experienced developers use to write clean, maintainable, and efficient code. These practices will make you a better programmer and prepare you for real-world development. By the end of this lesson, you'll understand how to write professional-quality JavaScript.

Why Best Practices Matter

Best practices are guidelines that help you write better code. They make code:

  • Easier to Read: Other developers (and future you!) can understand it quickly
  • Easier to Maintain: Bugs are easier to find and fix
  • More Reliable: Code is less likely to break or produce unexpected results
  • More Secure: Vulnerable code patterns are avoided
  • More Efficient: Performance problems are prevented
  • Team Compatible: Everyone follows the same standards

Naming Conventions

Use clear, descriptive names for variables, functions, and classes. Good names make code self-documenting:

Good Naming Examples

// Good naming - clear and descriptive let userEmail = 'user@example.com'; let isLoggedIn = true; let totalPrice = 99.99; function calculateTotalPrice(items) { return items.reduce((sum, item) => sum + item.price, 0); } class UserProfile { constructor(name, email) { this.name = name; this.email = email; } }

Bad Naming Examples

// Bad naming - unclear and confusing let x = 'user@example.com'; let a = true; let p = 99.99; function calc(arr) { return arr.reduce((s, i) => s + i.p, 0); } class u { constructor(n, e) { this.n = n; this.e = e; } }

Naming Conventions

  • camelCase for variables and functions: getUserName, isVisible
  • PascalCase for classes: UserProfile, ApiClient
  • UPPER_SNAKE_CASE for constants: MAX_ATTEMPTS = 3
  • Use meaningful prefixes:
    • is for booleans: isActive, isValid
    • has for booleans: hasChildren, hasPermission
    • get for getters: getUsername
    • set for setters: setPassword

Code Organization

Organize your code into logical sections. This makes projects easy to navigate and maintain:

Recommended File Structure

project/ ├── index.html // Main HTML file ├── styles/ │ └── main.css // All CSS ├── js/ │ ├── main.js // Entry point │ ├── utils/ │ │ ├── validators.js // Validation functions │ │ └── formatters.js // String/date formatters │ ├── modules/ │ │ ├── user.js // User management │ │ └── api.js // API calls │ └── components/ │ ├── header.js // Header component │ └── footer.js // Footer component └── README.md // Documentation

Single Responsibility Principle

Each function should do ONE thing and do it well:

// Good - each function has one responsibility function validateEmail(email) { return email.includes('@') && email.includes('.'); } function validatePassword(password) { return password.length >= 8; } function validateUser(email, password) { return validateEmail(email) && validatePassword(password); } // Bad - function does too much function validate(email, password) { // Handles both email AND password validation if (!email.includes('@')) return false; if (email.length === 0) return false; if (password.length < 8) return false; if (!password.includes('special')) return false; // Many more checks... return true; }

Keep Functions Small and Focused

Functions should be small (10-20 lines max). Smaller functions are easier to understand and test:

// Good - small, focused function function formatUserName(firstName, lastName) { return `${firstName} ${lastName}`.trim(); } // Bad - function does too much function processUser(data) { // Validates data if (!data.firstName) return null; if (!data.lastName) return null; // Formats data let name = `${data.firstName} ${data.lastName}`.trim(); // Saves to database saveToDatabase(name); // Sends email sendEmail(name); // Logs activity logActivity(name); return name; }

Variable Declaration Best Practices

Use const by Default

// Good - use const for values that don't change const MAX_ATTEMPTS = 3; const USER_ROLES = ['admin', 'user', 'guest']; const user = { name: 'John', email: 'john@example.com' }; // Use let only when variable will be reassigned let counter = 0; counter++; counter++; // Avoid var - it has confusing scope rules var oldStyle = 'outdated'; // Don't use this

Why Prefer const?

  • Prevents Accidents: Can't accidentally reassign the variable
  • Shows Intent: Makes it clear the value won't change
  • Better Performance: JavaScript engines can optimize const better
  • Encourages Good Design: Leads to more predictable code

Comments and Documentation

Write comments for complex logic, but keep them brief and useful:

Good Comments

// Calculate compound interest for monthly deposits // Formula: A = P(1 + r/n)^(nt) function calculateCompoundInterest(principal, rate, years) { const n = 12; // compounds monthly return principal * Math.pow(1 + rate / n, n * years); } // Filter users who haven't logged in for 30+ days const inactiveUsers = users.filter(user => { const daysSinceLogin = (Date.now() - user.lastLogin) / (1000 * 60 * 60 * 24); return daysSinceLogin > 30; });

Bad Comments

// Add one to counter let counter = 1; counter++; // Now counter is 2 // Loop through all items for (let i = 0; i < items.length; i++) { // Add item to list list.push(items[i]); }

Error Handling

Always handle errors gracefully. Don't let your program crash unexpectedly:

// Good - handles errors function fetchUserData(userId) { try { const user = getUserFromDatabase(userId); if (!user) { throw new Error(`User ${userId} not found`); } return user; } catch (error) { console.log('Error fetching user:', error.message); return null; } } // Bad - ignores errors function fetchUserData(userId) { const user = getUserFromDatabase(userId); return user; }

Input Validation

Always validate user input before using it. Never trust user data:

// Good - validates input function createUser(email, password) { // Validate email if (!email || typeof email !== 'string') { throw new Error('Invalid email'); } // Validate password if (!password || password.length < 8) { throw new Error('Password must be at least 8 characters'); } // Safe to use validated data return saveUser(email, password); } // Bad - uses input without validation function createUser(email, password) { return saveUser(email, password); // What if email or password are wrong type? }

Performance Best Practices

Cache DOM Queries

// Good - query once, use multiple times const button = document.getElementById('myButton'); button.addEventListener('click', handleClick); button.style.color = 'blue'; // Bad - queries DOM each time document.getElementById('myButton').addEventListener('click', handleClick); document.getElementById('myButton').style.color = 'blue';

Avoid Nested Loops

// Good - O(n) complexity const userIds = users.map(user => user.id); const activeUserIds = userIds.filter(id => isActive(id)); // Bad - O(n²) complexity - much slower! for (let i = 0; i < users.length; i++) { for (let j = 0; j < otherUsers.length; j++) { if (users[i].id === otherUsers[j].id) { // Do something } } }

Use Array Methods

// Good - modern, clean, fast const adults = users .filter(user => user.age >= 18) .map(user => user.name); // Bad - verbose, harder to read const adults = []; for (let i = 0; i < users.length; i++) { if (users[i].age >= 18) { adults.push(users[i].name); } }

Security Best Practices

Don't Use eval()

// Good - parse JSON properly const data = JSON.parse(jsonString); // Bad - eval is dangerous! const data = eval('(' + jsonString + ')'); // NEVER do this

Sanitize User Input

// Good - escape HTML special characters function escapeHtml(text) { const div = document.createElement('div'); div.textContent = text; return div.innerHTML; } const userName = escapeHtml(userInput); element.innerHTML = `Hello, ${userName}!`; // Bad - allows HTML injection attacks element.innerHTML = `Hello, ${userInput}!`;

Testing and Debugging

  • Test happy paths and edge cases - Test both normal and unusual inputs
  • Use console methods effectively - console.log(), console.error(), console.table()
  • Use browser DevTools - Debugger, Network tab, Console
  • Write unit tests - Test individual functions in isolation
  • Check browser compatibility - Test in different browsers

Code Style and Formatting

  • Use consistent indentation - 2 or 4 spaces (pick one and stick with it)
  • Use semicolons - Prevents unexpected behavior
  • Use meaningful whitespace - Separate logical sections
  • Keep lines reasonable length - Under 80-100 characters when possible
  • Use a linter - Tools like ESLint catch style issues automatically

Summary of JavaScript Best Practices

  • Use clear, descriptive names - Make code self-documenting
  • Keep functions small and focused - One responsibility per function
  • Use const by default - Only use let when variable will change
  • Handle errors gracefully - Use try-catch and never ignore errors
  • Validate all input - Never trust user data
  • Write meaningful comments - Explain WHY, not WHAT
  • Organize code logically - Use modules and folder structure
  • Test thoroughly - Write tests for critical functionality
  • Performance matters - Cache, avoid nested loops, use efficient algorithms
  • Security is critical - Sanitize input, avoid eval(), follow secure coding practices
